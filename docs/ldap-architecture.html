1. Overview — What We're Building

Goal:

Add LDAP as the fifth protocol to the fuzzer, with ~150 scenarios covering authentication attacks, search filter injection, BER encoding violations, AD-specific abuse, transport security (LDAPS/StartTLS), and CVE reproductions.

Why LDAP Fuzzing Matters

Active Directory is the backbone of enterprise networks. LDAP is the primary protocol for querying and modifying AD. Attackers use LDAP for Kerberoasting, credential relay, privilege escalation, and domain enumeration. A fuzzer that simulates an AD domain controller lets you test firewalls, LDAP proxies, and security tools without a real AD environment.

Capabilities

Client Mode

Sends malformed LDAP messages to a target server (real AD, OpenLDAP, or any LDAP implementation) to test how it handles protocol violations.

Server Mode (AD Simulation)

Acts as a fake Active Directory domain controller. Responds to LDAP queries with realistic AD data (users, groups, SPNs, GPOs). Tests LDAP clients and security tools.

Local Target

Self-contained mode: the fuzzer spawns both client and server internally. No external AD needed for testing.

Three Transports

Plain LDAP on port 389, LDAPS (TLS-wrapped) on port 636, and StartTLS upgrade on port 389. All hand-built, all fuzzable.

2. LDAP Protocol Primer

LDAP (Lightweight Directory Access Protocol) is a TCP-based protocol for reading and writing directory data. Every message is encoded in ASN.1 BER (Basic Encoding Rules) and wrapped in a common envelope:

LDAP Operations at a Glance

Operation Tag Direction Purpose
BindRequest 0x60 Client → Server Authenticate (simple password or SASL/NTLM)
BindResponse 0x61 Server → Client Authentication result (success/invalidCredentials)
UnbindRequest 0x42 Client → Server Graceful disconnect (no response)
SearchRequest 0x63 Client → Server Query directory entries by filter
SearchResultEntry 0x64 Server → Client One matching entry (DN + attributes)
SearchResultDone 0x65 Server → Client End of search results with status code
ModifyRequest 0x66 Client → Server Add/delete/replace attributes on an entry
AddRequest 0x68 Client → Server Create a new directory entry
DelRequest 0x4A Client → Server Delete an entry by DN
CompareRequest 0x6E Client → Server Test if an attribute has a specific value
AbandonRequest 0x50 Client → Server Cancel an in-progress operation
ExtendedRequest 0x77 Client → Server StartTLS, WhoAmI, PasswordModify, AD FastBind

3. BER/ASN.1 Wire Format

Every LDAP byte on the wire follows the Tag-Length-Value (TLV) pattern. This is what lib/ldap-packet.js builds.

(1+ bytes)"] --> L["Length
(1+ bytes)"] --> V["Value
(N bytes)"] style T fill:#ff7b72,color:black,stroke:none style L fill:#7ee787,color:black,stroke:none style V fill:#a5d6ff,color:black,stroke:none

Tag Byte Structure

// Tag byte layout: [class:2][constructed:1][number:5] // // Class: 00 = Universal 01 = Application // 10 = Context 11 = Private // // Constructed: 0 = Primitive 1 = Constructed (contains TLVs) // // Examples: 0x30 = Universal + Constructed + 16 = SEQUENCE 0x02 = Universal + Primitive + 2 = INTEGER 0x04 = Universal + Primitive + 4 = OCTET STRING 0x01 = Universal + Primitive + 1 = BOOLEAN 0x0A = Universal + Primitive + 10 = ENUMERATED 0x60 = Application + Constructed + 0 = BindRequest 0x42 = Application + Primitive + 2 = UnbindRequest 0xA0 = Context + Constructed + 0 = Context [0] (AND filter / SASL creds) 0x87 = Context + Primitive + 7 = Context [7] (present filter)

Length Encoding

// Short form (value < 128): single byte 0x05 // length = 5 // Long form (value ≥ 128): 0x80 + count, then count bytes 0x81 0x80 // length = 128 (1 extra byte) 0x82 0x01 0x00 // length = 256 (2 extra bytes) 0x84 0xFF 0xFF 0xFF 0xFF // length = 4GB (fuzz: overflow!)

How Scenarios Are Routed

The UnifiedClient reads the scenario's category prefix to dispatch to the correct engine:

function isLdapScenario(scenario) { return scenario.category.length === 2 && scenario.category[0] === 'L' && scenario.category[1] >= 'A' && scenario.category[1] <= 'Z'; } // Dispatch order in runScenario(): if (isLdapScenario(scenario)) return ldapClient.runScenario(scenario); if (isTcpScenario(scenario)) return this._runRawTCPScenario(scenario); if (isQuicScenario(scenario)) return quicClient.runScenario(scenario); if (isH2Scenario(scenario)) return this._runH2Scenario(scenario); return this._runTLSScenario(scenario); // default

5. New Modules

5.1 ldap-packet.js — BER/ASN.1 Wire Format Builder

Analogous to record.js + handshake.js for TLS, or quic-packet.js for QUIC. Builds every LDAP message as raw Buffer bytes.

LayerFunctionsPurpose
BER Primitives berSequence(), berInteger(), berOctetString(), berBoolean(), berEnumerated(), berContextTag(), berApplication() Low-level TLV encoding. Everything else is built on these.
LDAP Message Builders buildBindRequest(), buildSearchRequest(), buildModifyRequest(), buildAddRequest(), buildDelRequest(), buildCompareRequest(), buildExtendedRequest(), buildAbandonRequest(), buildUnbindRequest() Complete LDAP messages ready to write to socket.
LDAP Response Builders buildBindResponse(), buildSearchResultEntry(), buildSearchResultDone(), buildModifyResponse(), buildExtendedResponse() For AD simulation server to send back to clients.
Filter Builder buildFilter() Recursive: AND/OR/NOT/eq/sub/gte/lte/present/extensibleMatch.
Parser parseLDAPMessage() Extract messageID, protocolOp tag, resultCode from response bytes.
Fuzz Helpers berMalformedLength(), buildNestedDepthBomb(), buildOversizeString(), buildGarbageMessage(), buildPartialMessage() Intentionally broken BER for testing error handling.

5.2 ldap-scenarios.js — Scenario Definitions

~150 scenarios across 12 categories (LA through LL). Same structure as existing scenario files.

5.3 ldap-fuzzer-client.js — Client Engine

TCP-based client following the FuzzerClient pattern. Connects via net.connect() (plain) or tls.connect() (LDAPS), sends raw BER bytes, parses responses.

sequenceDiagram participant S as Scenario participant C as LdapFuzzerClient participant T as Target Server S->>C: actions = scenario.actions() C->>T: TCP connect (port 389) loop For each action alt type = 'send' C->>T: socket.write(BER bytes) Note over C,T: e.g. BindRequest, SearchRequest else type = 'recv' T->>C: raw BER response C->>C: parseLDAPMessage() + _describeLdapResponse() else type = 'starttls' C->>T: ExtendedRequest (StartTLS OID) T->>C: ExtendedResponse (success) C->>C: Upgrade socket to TLS else type = 'delay' C->>C: sleep(ms) end end C->>C: Grade result (PASS/FAIL/WARN) style C fill:#b45309,color:white

5.4 ldap-fuzzer-server.js — AD Simulation Server

A fake Active Directory domain controller. Listens on TCP, parses incoming LDAP messages, responds with realistic AD data.

7. Scenario Categories & Coverage

12 categories, ~150 total scenarios. Categories use the L prefix (LA through LL), consistent with TLS (A-Y), HTTP/2 (AA-AO), QUIC (QA-QL), TCP (RA-RH).

LA — Authentication Attacks

~15 scenarios. Simple bind, anonymous bind, null bind, SASL/NTLM, brute force, protocol version downgrade, unauthenticated bind (MS AD default behavior).

LB — Search Filter Injection

~12 scenarios. Wildcard abuse, nested filter depth bombs, malformed filters, extensibleMatch chaining, null bytes, oversized filters.

LC — BER/ASN.1 Violations

~18 scenarios. Truncated lengths, indefinite encoding, wrong tags, depth bombs (1000-level nesting), split messages, non-minimal encodings.

LD — Protocol Sequence Violations

~10 scenarios. Search before bind, request after unbind, duplicate message IDs, role reversal (client sends server messages).

LE — Resource Exhaustion

~10 scenarios. Connection flood, unlimited search, paged results abuse, 10MB payload, connection churn.

LF — LDAPS/StartTLS Transport

~12 scenarios. LDAPS on 636, StartTLS upgrade, downgrade attack, double StartTLS, expired/self-signed certs, channel binding, TLS renegotiation.

LG — AD-Specific Attacks

~15 scenarios. SPN enumeration (Kerberoasting), AS-REP roasting, LAPS password read, gMSA extraction, delegation abuse, GPO enumeration, DCShadow checks, SID History injection.

LH — Operation Fuzzing

~10 scenarios. Missing required attributes, invalid operation codes, binary values, non-leaf delete, empty modification lists.

LI — Extended Operations

~8 scenarios. WhoAmI, PasswordModify, Cancel, unknown OID, AD FastBind, oversized extended value.

LJ — Server-to-Client Attacks

~12 scenarios. Server sends malformed responses, referral injection, oversized entries, infinite result stream, wrong messageID, BER depth bomb in response.

LK — CVE Reproductions

~10 scenarios. Zerologon (LDAP vector), noPac, ProxyLogon recon, Log4Shell JNDI:LDAP, NTLM MIC bypass, PetitPotam LDAP coercion.

LL — Connectivity & Baseline

~5 scenarios. RootDSE query, schema discovery, full bind/search/unbind lifecycle, feature detection.

Transport Selection in Scenarios

// Scenario controls transport via the 'connect' action: actions: () => [ { type: 'connect', mode: 'plain' }, // net.connect() to port 389 { type: 'connect', mode: 'tls' }, // tls.connect() to port 636 { type: 'connect', mode: 'starttls' }, // net.connect(389), then upgrade // If no 'connect' action, defaults to plain TCP { type: 'send', data: buildBindRequest(...) }, { type: 'recv', timeout: 5000 }, ]
StartTLS Upgrade:

Node.js supports upgrading an existing net.Socket to TLS via tls.connect({ socket: existingSocket }). The client sends the StartTLS ExtendedRequest on the plain connection, waits for a success response, then wraps the socket in TLS. All subsequent LDAP messages are encrypted.

10. Integration with Existing Codebase

Key Insight:

The fuzzer's architecture is protocol-agnostic. Adding LDAP means creating 4 new files and making surgical edits to 12 existing files. No architectural changes needed.

Critical Integration: isH2Scenario() Fix

The current HTTP/2 detection treats all two-letter categories that aren't Q* or R* as HTTP/2. After adding LDAP (L*), this must exclude L* to prevent LDAP scenarios routing to the HTTP/2 engine.

// BEFORE (broken for LDAP): function isH2Scenario(s) { return s.category.length === 2 && !isQuicScenario(s) && !isTcpScenario(s); } // AFTER (correct): function isH2Scenario(s) { return s.category.length === 2 && !isQuicScenario(s) && !isTcpScenario(s) && !isLdapScenario(s); }
