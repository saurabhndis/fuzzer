A deep dive into the design, flows, and engines of the multi-protocol fuzzing suite.

1. System Architecture

The tool uses a Node.js core with an Electron-based GUI. It abstracts protocol complexities into a Unified Engine layer that manages execution across TLS, HTTP/2, QUIC, and Raw TCP.

3. Well-Behaved vs. Fuzzed Scenarios

The fuzzer operates by running two types of tests for every protocol. This dual-path approach ensures that "AS EXPECTED" results are validated against true protocol compliance.

Well-Behaved (QA/Baseline)

Uses standard libraries (OpenSSL, Node.js tls/http2) to establish a perfect, RFC-compliant connection. Validates that the target is reachable and functioning under normal conditions.

Fuzzed (Protocol Attack)

Uses custom byte-level builders to craft illegal or edge-case packets. Manually manipulates handshake states, record lengths, and frame types to probe the target's error handling.

8. Application Layer Payloads

Beyond protocol headers, the tool tests the security engine's ability to detect and block malicious content nested within the established encrypted streams.

Virus & Malware

Injects EICAR test strings and known malware signatures (Spyware/Trojans) into TLS and HTTP/3 GET/POST bodies.

Sandbox JS

Sends malicious JavaScript designed to trigger sandbox heuristics (Heap sprays, environment probes, obfuscated evasion).

URL Categories

Uses 27 categories of PAN-OS domains in SNI headers to test URL filtering policy enforcement.

PQC Evasion

Combines 2KB ML-KEM (Kyber) payloads with SNI fragmentation to hide malicious intent from DPI.

9. How Node.js Powers This Project

Node.js is the runtime foundation for the entire fuzzer. Understanding its architecture explains why this tool can fuzz thousands of connections concurrently on a single machine without thread-per-connection overhead.

Event Loop & Non-Blocking I/O

Node.js runs JavaScript on Google's V8 engine atop libuv, which provides an event loop and cross-platform async I/O. Every network operation — TCP connect, TLS handshake, UDP send — is non-blocking. The fuzzer issues a socket.write() and immediately proceeds to the next action; a 'data' event fires when the target responds. This is what allows a single worker process to manage hundreds of in-flight fuzz connections simultaneously.

Built-in Modules Used Extensively

The fuzzer relies heavily on Node.js built-in modules rather than third-party packages:

net

Raw TCP sockets for TLS record-level fuzzing. Every TLS fuzz scenario creates a net.Socket and writes hand-crafted bytes.

tls

Well-behaved baseline connections. Used only for "golden path" tests to confirm the target is alive and accepts valid TLS.

http2

Both compliant HTTP/2 sessions (baseline) and as a transport for frame-level injection after ALPN negotiation.

dgram

UDP sockets for stateless QUIC Initial packet fuzzing. Sends raw QUIC packets without any handshake state.

child_process

Fork-based worker pool. Also spawns openssl s_client for baseline testing and QUIC post-handshake injection.

crypto

Self-signed X.509 certificate generation (RSA 2048-bit) and QUIC Initial packet AEAD key derivation.

12. TLS Fuzzing Layer — Deep Dive

TLS fuzzing is the largest and most granular layer, with 26 scenario categories (A–Z) covering every phase of the TLS handshake and record layer.

How It Works

Unlike conventional TLS clients that use OpenSSL or Node's tls module, the fuzzer opens a raw TCP socket (net.Socket) and writes hand-crafted TLS records byte-by-byte. This bypasses all library-level validation, allowing the fuzzer to produce protocol states that no compliant implementation would generate.

graph TD subgraph "TLS Fuzzer Internals" SC[scenarios.js: 26 categories] -->|actions| FC[FuzzerClient] FC -->|net.Socket| TCP[Raw TCP Connection] subgraph "Packet Construction (Pure JS)" HS[handshake.js: ClientHello, ServerHello, Finished, CertificateVerify] REC[record.js: wrapRecord, buildAlert, buildCCS] CONST[constants.js: cipher suites, versions, alert codes] X509[x509.js: Certificate field mutation] end FC --> HS FC --> REC FC --> CONST FC --> X509 TCP -->|raw bytes| Target((Target)) Target -->|response bytes| FC FC -->|parse TLS records| Verdict{PASS / FAIL} end

Category Breakdown

A–B: Handshake State Machine

Sends handshake messages in illegal order (e.g., Finished before ServerHello). Tests that the target's state machine rejects out-of-sequence transitions.

C–D: Parameter & Alert Mutation

Mutates version fields, random bytes, session IDs. Injects alerts at wrong handshake phases to confuse the state machine.

F–G: Record Layer & CCS

Fragments records across TCP segments, sends zero-length records, injects ChangeCipherSpec at illegal points.

H–R: Extension & Field Fuzzing

Malformed extensions, duplicate extensions, oversized SNI, ALPN confusion, 0-RTT abuse, ClientHello field mutations.

I–J: CVE & PQC

Known vulnerability signatures (Heartbleed, POODLE, ROBOT). Post-Quantum key exchange bloat attacks using ML-KEM payloads.

W–Y: Certificate Fuzzing

X.509 field mutations (oversized CN, invalid SAN, expired certs, wrong key usage), certificate chain attacks, client cert abuse.

14. QUIC Fuzzing Layer — Deep Dive

QUIC fuzzing is the most architecturally complex layer because QUIC encrypts from the very first packet. The fuzzer uses three distinct engines depending on the test depth required.

Three-Engine Architecture

Engine Selection Logic

The QuicFuzzerClient selects the appropriate engine per scenario based on the required test depth:

EngineWhen UsedCapabilitiesLimitations
Stateless UDP Initial packet fuzzing (QA, QB, QF) Full control over every byte. No handshake needed. Fastest. Cannot reach post-handshake state. No encryption beyond Initial.
OpenSSL Post-handshake app payload injection (QJ CVEs) Full TLS 1.3 handshake. Application-layer payload delivery. Requires OpenSSL 3.6+. Limited frame-level control. External process.
Quiche Deep stateful fuzzing (QC, QD, QE, QG–QK) Full QUIC state machine. Frame-level mutations. HTTP/3 streams. Requires optional @currentspace/http3 dependency.

15. Worker Thread Pool — Deep Dive

The fuzzer uses Node.js child_process.fork() to create a pool of worker processes. This is not the worker_threads module — each worker is a full, isolated Node.js process with its own V8 heap, event loop, and TCP/UDP sockets.

Why Processes, Not Threads?

Process Isolation (chosen)

Crash isolation: If a scenario causes a segfault in a native addon (raw-socket, quiche), only that worker dies. The primary and other workers continue.

Socket handles: TCP socket handles can be passed from primary to workers via IPC sendHandle. Not possible with worker_threads.

Clean state: Each worker starts fresh — no leaked TLS contexts, no stale connection state between scenarios.

Worker Threads (not used)

Shared memory: Would allow SharedArrayBuffer between workers, but the fuzzer doesn't need shared state — each scenario is independent.

Lower overhead: Threads are lighter, but the fuzzer's bottleneck is network I/O, not process creation. Fork cost is negligible.

Worker Pool Variants

The same work-stealing pattern is used across all three interfaces, with slight variations in how workers are spawned and how results are collected:

ContextPrimaryWorkerCommunicationConcurrency
CLI (client.js) Same file (isPrimary check) Same file (isWorker path) process.send() / process.on('message') N workers (default: CPU count)

Work-Stealing Queue Pattern

The primary maintains a FIFO queue of scenarios. Workers pull work when ready — there is no pre-assignment. This naturally load-balances across workers: fast scenarios (e.g., a rejected handshake that returns an alert in 10ms) free up the worker immediately for the next task, while slow scenarios (e.g., a timeout after 5 seconds) don't block other workers.

PCAP & Logging in Multi-Worker Mode

Each worker writes to its own PCAP file (output.pcap.worker-<PID>) to avoid file contention. Log events are streamed from workers to the primary via IPC and rendered in order. In Electron mode, log events are forwarded to the renderer for real-time display.
