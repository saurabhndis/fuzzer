Phase 1: BER/ASN.1 Wire Format

lib/ldap-packet.js — Foundation layer, no dependencies on other new code

This is the equivalent of record.js + handshake.js for TLS. Every LDAP byte on the wire is built here.

Step 1.1: BER Encoding Primitives

// Core TLV functions that everything else builds on function berLength(len) { // Short form: single byte if len < 128 // Long form: 0x81 XX for len < 256, 0x82 XX XX for len < 65536 } function berSequence(contents) { // Tag 0x30 + berLength(contents.length) + contents } function berInteger(value) { // Tag 0x02 + length + signed big-endian bytes } function berOctetString(buf) { // Tag 0x04 + length + raw bytes } function berBoolean(val) { // Tag 0x01 + 0x01 + (0xFF or 0x00) } function berEnumerated(val) { // Tag 0x0A + length + value bytes } function berContextTag(n, contents, constructed) { // constructed: tag = 0xA0 + n primitive: tag = 0x80 + n } function berApplication(n, contents, constructed) { // constructed: tag = 0x60 + n primitive: tag = 0x40 + n }

Step 1.3: Search Filter Builder

function buildFilter(spec) { // Recursive filter construction // { present: 'objectClass' } → context[7] primitive (0x87) // { eq: { attr: 'cn', value: 'admin' } } → context[3] constructed (0xA3) // { and: [filter1, filter2] } → context[0] constructed (0xA0) // { or: [filter1, filter2] } → context[1] constructed (0xA1) // { not: filter } → context[2] constructed (0xA2) // { sub: { attr, initial, any, final } } → context[4] (0xA4) // { gte: { attr, value } } → context[5] (0xA5) // { lte: { attr, value } } → context[6] (0xA6) // { ext: { rule, attr, value, dn } } → context[9] (0xA9) }

Step 1.4: Parser + Fuzz Helpers

  • 🔎

    parseLDAPMessage(buffer)

    Minimal parser: extract messageID, protocolOp tag byte, resultCode (for responses), matchedDN, diagnosticMessage, and totalLength (bytes consumed, for TCP framing).

  • 💣

    Fuzz helpers

    berMalformedLength(type) — 'truncated', 'zero', 'overflow' (0x84 FFFFFFFF), 'indefinite' (0x80 without terminator).
    buildNestedDepthBomb(depth) — 1000-level nested SEQUENCEs.
    buildOversizeString(size), buildGarbageMessage(size), buildPartialMessage(msg, cutAt).

class LdapFuzzerClient { constructor(opts) { this.host = opts.host || 'localhost'; this.port = opts.port || 389; // 389 for LDAP, 636 for LDAPS this.timeout = opts.timeout || 5000; this.delay = opts.delay || 100; this.logger = opts.logger; } // Main execution method — same contract as FuzzerClient.runScenario() async runScenario(scenario) { ... } async runScenarios(scenarios) { ... } // Connection methods _connectLDAP() { /* net.connect(port, host) */ } _connectLDAPS() { /* tls.connect({ port: 636, host, ... }) */ } _connectStartTLS() { /* plain connect, send ExtReq, tls.connect({socket}) */ } // Response handling _describeLdapResponse(data) { // Parse BER, return human-readable string: // "BindResponse(success)", "BindResponse(invalidCredentials)" // "SearchResultDone(noSuchObject)", "protocolError" } }

Action Types

ActionParametersBehavior
senddata: Buffer, label: stringWrite BER bytes to TCP socket
recvtimeout: msRead response from socket, parse LDAP message
delayms: numberSleep
connectmode: 'plain'|'tls'|'starttls'Establish connection with specified transport
starttls—Send StartTLS ExtendedRequest + upgrade socket
close—Close TCP connection
TCP Framing:

Unlike QUIC (one UDP datagram = one message), LDAP messages can split across TCP segments. The client must accumulate data and parse BER length prefixes to know when a complete message has arrived. Use parseLDAPMessage(buffer).totalLength to determine framing boundaries.

Phase 4: AD Simulation Server

lib/ldap-fuzzer-server.js — Depends on Phase 1 (ldap-packet.js)

Class: LdapFuzzerServer

class LdapFuzzerServer { constructor(opts) { this.port = opts.port || 389; this.tlsPort = opts.tlsPort || 636; this.hostname = opts.hostname || 'localhost'; this.adConfig = this._defaultADConfig(); } async start() { /* net.createServer(), listen on port 389 */ } async startTLS() { /* tls.createServer(), listen on port 636 */ } // Connection handler — parses incoming BER, dispatches to operation handler _handleConnection(socket) { ... } // Operation handlers _handleBind(socket, msgId, body) { /* check credentials, respond */ } _handleSearch(socket, msgId, body) { /* query in-memory entries */ } _handleSearchRootDSE(socket, msgId) { /* return AD RootDSE */ } _handleModify(socket, msgId, body) { /* log + respond */ } _handleAdd(socket, msgId, body) { /* log + respond */ } _handleDelete(socket, msgId, body) { /* log + respond */ } _handleExtended(socket, msgId, body){ /* StartTLS, WhoAmI, etc. */ } // For fuzz scenarios (category LJ — server-to-client attacks) async runScenario(scenario) { /* execute serverHandler or actions */ } }

AD Simulation Fidelity

Design principle:

The server does not implement full Active Directory. It provides just enough to: (1) accept/reject binds, (2) return AD-identifying RootDSE, (3) serve simulated entries for SPN/LAPS/gMSA queries, (4) handle StartTLS upgrade. Complex behaviors like replication, Kerberos, or group policy processing return unwillingToPerform.

Critical: isH2Scenario() Must Be Updated

Without this fix, all LDAP scenarios (category L*) will be misrouted to the HTTP/2 engine. This is the single most important correctness concern. The fix is a one-line addition: && !isLdapScenario(scenario).

Phase 6: Local Target Support

well-behaved-server.js + well-behaved-client.js

When "Local Target" is enabled in the GUI, the fuzzer spawns a compliant counterpart:

  • Client mode fuzzing: well-behaved-server.js starts a minimal LDAP server that accepts bind and returns basic search results.
  • Server mode fuzzing: well-behaved-client.js connects and sends an anonymous bind to trigger the fuzzed server's response path.

New Methods

FileMethodBehavior
well-behaved-server.js startLdap() Create net.Server, parse incoming LDAP messages, respond with BindResponse(success) and basic SearchResultEntry/Done.
well-behaved-client.js connectLdap() Open TCP to server, send anonymous BindRequest, wait for response. Retry on ECONNREFUSED (server may still be starting).

11. Risks & Mitigations

RiskSeverityMitigation
isH2Scenario() not updated — LDAP scenarios route to HTTP/2 engine HIGH Add && !isLdapScenario(scenario) to both unified-client.js and unified-server.js. Test with a single LDAP scenario before bulk integration.
BER framing over TCP — messages split across segments MEDIUM Accumulate data in buffer, use BER length prefix to detect complete messages. Same pattern as TLS record framing in parseRecords().
StartTLS socket upgrade fails MEDIUM Use Node.js tls.connect({ socket: existingSocket }) which is the standard approach. Test against OpenLDAP with StartTLS first.
AD simulation not realistic enough LOW Focus on RootDSE fidelity (OIDs, capabilities) and key entries (SPNs, LAPS). Iteratively add entries as needed. Complex features return unwillingToPerform.
Category collision with future protocols LOW L* prefix is unique. Document in architecture.html that L is reserved for LDAP.
